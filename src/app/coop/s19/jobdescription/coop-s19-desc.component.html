Overview and Main Tasks

As a Unitron Web Software Developer you work alongside other developers to implement web-based functionality. Unitron developers work under a strong and well thought out development architecture and process. In addition, Unitron developers work in an agile, collaborative and transparent working environment.

Main tasks include:

  • Developing and analyzing web-based software solutions for the hearing aid markets
  • Troubleshooting software/system issues as needed
  • Participating in agile development
  • Work closely with other teams such as marketing to ensure the final product adheres to the set requirements

Required Skills

As a Unitron Web Software Developer you are required to bring the standard technical skills for a web development job. Experience with Node.js, Angular 2, Javascript, HTML, CSS, and SQL server is required to be able to work on all facets of the various projects.

Personally, my experience in JS, HTML, CSS and Node.js came in personal projects where I created web apps. I also had previously used SQL, but my SQL skills were reinforced when it was needed in my CIS*2750 class. It is also important to mention that Unitron provides great resources to improve these skills and bring you up to a level where you can work efficiently.

My Day-to-Day Experience

Working as a developer at Unitron was a very interesting experience. I learned very quickly that as a new co-op its about trust. Completing tasks at highest quality possible in the allotted time allows other developers to entrust you more with large and more complex tasks. From my experience, there are no any specific tasks set aside just for Unitron co-ops. Instead, as co-ops grow as developers they are given more and more complex tasks.

Another interesting part of working at Unitron is that you work closely with marketing team to ensure that requirements are met and to communicate any potential changes in the requirements. In addition, as an agile team, we had frequent stand-ups where we discussed what we completed before the stand-up as well as upcoming tasks.

As developers we had frequent stand-up meetings in order to ensure that we know what each developer is doing.
